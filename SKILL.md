# 生鲜商品视频策划技能

## 固定配置

```
视频输出根目录：~/freshVedioPlanning/
视频规格：时长 ≤15s | 分辨率 720p | 比例 9:16
即梦CLI命令：~/.local/bin/dreamina multimodal2video（全能参考模式，支持多图）
```

## 触发条件

每次任务开始前，**逐项收集商品信息**（见下方清单），如有缺失立即暂停并请求用户提供。不得跳过，不得自行搜索或推理补充。

## 第一步：收集商品信息

向用户收集以下字段（其中 `product_analysis` 为选填，其余为必填）：

| 字段 | 说明 | 示例 |
|------|------|------|
| `product_name` | 商品名称 | 美汁源益生元凤梨百香果复合果汁饮料 |
| `sku` | 商品SKU编号 | 556127 |
| `selling_points` | 商品核心卖点（多条，用换行分隔） | 30%膳食纤维、凤梨+百香果双果、0脂 |
| `user_demands` | 用户需求/使用场景（多条） | 餐后解腻、肠道健康、下午茶 |
| `hot_topics` | 当前热点/季节机会（可选填） | 轻体季、健康饮品 |
| `product_analysis` | 商品分析（选填，可粘贴类目表现/搜索趋势/行业调研/价格力/竞争力等结论） | 细分类目表现、搜索热度、行业分析、价格力、竞争力、总分 |
| `main_image` | 商品主图本地路径 | ~/freshVedioPlanning/.../main_image.jpg |
| `detail_images` | 商详图片路径列表（至少1张） | ~/freshVedioPlanning/.../detail1.jpg |

**缺少任意必填字段 → 立即告知用户缺失项，停止后续流程。`product_analysis` 为选填，不作为阻断项。**

## 第二步：建立会话目录

```bash
python3 ~/.qclaw/skills/fresh-video-planning/scripts/init_session.py \
  --product "商品名称" --sku "sku编号" --root "~/freshVedioPlanning"
```

脚本输出本次 `run_NNN` 目录路径，后续步骤均在该路径下进行。

## 第三步：切入点分析（世界观构建）

将分析结果保存至会话目录的 `01_切入点分析.md`。

若提供了 `product_analysis`，先提取其中可直接用于创意表达的“机会信号”（类目竞争、季节趋势、价格带、人群认知），仅允许引用有明确依据的结论，禁止杜撰数据。

必须输出以下结构，不得省略：

### 3.1 产品诊断（5维）

围绕商品信息提取 5 个维度：

1. 原料（产地/品种/独特性）
2. 成分（健康数据/核心营养）
3. 颜色（天然色泽来源）
4. 香气（气味记忆点）
5. 工艺（处理方式/NFC/冷灌等）

### 3.2 卖点翻译（核心）

将每个卖点翻译成可视化角色或场景，禁止直白罗列参数：

- 原料 -> 产地奇观（庄园、矿区、果园、农场）
- 成分 -> 微观角色（粒子、晶体、能量线）
- 颜色 -> 巨物奇观（切开瞬间、渐变流体）
- 香气 -> 可视化氛围（烟雾、光丝、雾气）
- 工艺 -> 游戏化流程（漂流、汇合、闯关）

新增"真实卖点提取"步骤（必须执行）：

1. 从主图、商详图、卖点文案中提取"可被证据支撑"的卖点清单（禁止凭空补充）。
2. 为每个卖点标注类型：  
   - A 类：产品外观/包装可直接验证  
   - B 类：原材料特性可直接验证  
   - C 类：功能或口感主张（需谨慎表达，不可伪造内部状态）
3. 仅允许 A/B 类卖点直接转镜头；C 类卖点通过场景结果、情绪反馈、抽象质感间接表达。
4. 若核心卖点属于"原材料特性"，分镜前两镜必须表现原材料特点，不得与产品内部形态融合。

**⚠️ 食材组合真实性约束（新增）：**
1. **原料图与成品图必须区分使用**：
   - 原料独立展示图（如石斛条、干枸杞）**只能用于"原材料特写"镜头**，不得作为成品内部食材的参考
   - 成品图（如炖好的烤梨）中如果没有该食材的真实出现，**禁止凭空添加**
2. **组合描述必须有真实图片支撑**：
   - "斜搭在..."、"叠加在..."、"浸润在..."等组合描述，必须有对应的真实拍摄图片
   - 若无真实组合图，必须分镜展示：先展示原料独立特写，再展示成品状态
3. **食材状态必须符合烹饪逻辑**：
   - 炖煮类产品的食材应是"浸润在汤汁中"的状态，不是"生硬叠加"或"干条斜搭"
   - 若原料图是干燥状态，成品中应呈现"炖煮后的软化状态"，不可直接复制干燥状态
4. **开盖内景证据闸门（强制）**：
   - 若没有同 SKU 的真实开盖内景图片，禁止生成“开盖后可见内部食材细节”镜头
   - 此时仅允许：外包装开盖动作 + 冷气/光影/香气隐喻，不得出现碗内具体食材组合细节
   - 若存在真实开盖内景图，必须在参考图中明确标注“开盖内景证据图”
5. **原料卖点优先镜头策略（与开盖闸门联动）**：
   - 当“开盖内景不明确”且核心卖点属于原料特性时，前 2-3 镜必须优先表现原料特征（产地、形态、纹理、处理过程）
   - 这类场景下，产品镜头应先做外包装/外观展示，再在后续镜头承接品牌落版
   - 禁止在前几镜通过想象性“碗内内容”去替代原料镜头

### 3.3 世界观构建（只保留 1 套）

基于上一步，构建一个 AI 原生世界观（只产出 1 套）：

- 世界设定：产品发生在什么奇观场域（例：味觉游乐场、原料冒险基地）
- 角色设定：原料/成分在世界中的角色身份
- 目标设定：产品如何作为“冒险终点/价值兑现”
- 核心反差：先破旧认知，再给新认知（例如“以为人工调色，实际天然来源”）

### 3.4 平台适配约束（抖音/即梦）

在分析末尾给出“本次必须遵守”的 4 条硬约束：

1. 前1秒必须出现视觉主体（果实/瓶身/动作）
2. 前3秒必须出现冲突或反差（尺度、空间、情绪）
3. 每4秒至少一次信息增量（新动作/新角度/新卖点）
4. 结尾2秒必须完成品牌落版 + 记忆锚点（标签/口号/动作）

## 第五步：生成分镜脚本（单套）

生成1套完整分镜脚本，保存至 `03_分镜脚本.md`。

若存在 `product_analysis`，分镜中的卖点文案与镜头重点应优先对齐“高置信分析结论”（如搜索高热词、已验证差异点、价格力优势）；未被分析支持的卖点，需降级为辅助表达，不可作为核心钩子。

### 5.1 分镜脚本生成逻辑

必须按以下顺序生成，且只产出 1 套可直接执行的后半段产品成片脚本：

1. **先做素材事实表**：逐张写明"文件名 -> 实际内容 -> 可用于哪一镜"，禁止凭文件名猜图。  
2. **再写世界观与质感基准**：给出一句核心世界观 + 画幅/质感红线。  
3. **再做参考图分工**：每镜至少声明 2 种角色（首帧锚定/主体纹理/色彩氛围/构图关系），避免"同一张图承担全部任务"。  
4. **按镜头逐个展开**：建议 4 镜（3-4-4-4 秒或近似），每镜必须含构图公式、光影公式、色彩公式。  
5. **每镜补齐执行参数**：必须写清镜头运动、时间事件（如 T+2.0s）、转场方式、对焦策略、本镜禁止项。  
6. **每镜必须带 Seedance Prompt**：镜头描述和最终提示词一一对应，避免"脚本写一套、提示词是另一套"。  
7. **最后补时间轴和素材分配汇总**：确保总时长 <= 15 秒，素材用途可追溯。  
8. **禁用不实内部镜头**：无法证实包装/杯内状态时，不得写拆封后内部细节。
9. **开盖不明确时的镜头优先级**：若开盖内景证据不足且卖点偏原料特性，镜头优先级固定为“原料特写 -> 原料处理/环境 -> 产品外观 -> 品牌落版”。

**开盖不明确 + 原料卖点 的推荐节奏（默认模板）**：
- Shot01（0-3s）：原料识别镜头（单一原料特写，建立记忆点）
- Shot02（3-7s）：原料处理/环境镜头（炖煮、浸润、蒸汽、产地氛围等）
- Shot03（7-11s）：产品外观与包装镜头（不展示未经证实的内景）
- Shot04（11-15s）：品牌落版与核心卖点收束

**⚠️ 动态节奏约束（新增，必须遵守）：**
1. **至少1个快节奏镜头**：15秒视频必须包含至少1个"快节奏"镜头（快推、急停、剧烈转场），禁止四个镜头都是"缓降"、"缓推"等平稳运动
2. **必须包含食欲感动态**：至少1个镜头必须包含"食欲感动态"，如热气升腾、汤汁流动、勺子舀起、筷子夹起、食材滴落等
3. **禁止全程静态定格**：除"微距特写"镜头外，禁止连续3个镜头以上使用"定格"、"静止"等静态运镜

**⚠️ 视觉冲击约束（新增，必须遵守）：**
1. **至少1个视觉奇观镜头**：15秒视频必须包含至少1个"视觉奇观"镜头，如尺度错位、空间反转、光影穿越、色彩爆发等
2. **世界观必须落地到镜头**：若设定了"仙山"、"山谷"等世界观，必须有至少1个镜头包含"环境氛围元素"（山林剪影、云雾、光影、色彩），不能只在文案层面提及
3. **开场3秒必须抓住注意力**：第一个镜头必须有明确的视觉冲突或动态，禁止"静态展示+缓慢推进"的开场

### 6.2 调用即梦CLI（一次性合并提交）

**⚠️ 关键：必须一次性合并所有镜头提示词，使用 multimodal2video 提交！禁止逐镜单独提交！**

**提交方式：** 将所有镜头的提示词按 Shot 顺序拼接后，生成两份文件：`即梦提示词_校验版.txt` 与 `即梦提示词_提交版.txt`。实际调用 `multimodal2video` 时，`--prompt` 必须使用 `即梦提示词_提交版.txt`，一次性提交。

**提交流程：**

1. **先读取分镜脚本** — `read_file` 读取 `03_分镜脚本.md`，逐镜提取完整 Prompt
2. **合并提示词** — 按 Shot 顺序拼接所有镜头的提示词（参考图描述+镜头参数+时间事件+禁止项+Prompt正文），作为一段完整文本
3. **一次性提交** — 调用 `multimodal2video`，使用所有镜头涉及的参考图 + 合并后的提示词

```bash
# 一次性提交所有镜头（示例）
~/.local/bin/dreamina multimodal2video \
  --image "<会话目录/参考图1.jpg>" \
  --image "<会话目录/参考图2.jpg>" \
  --image "<会话目录/参考图3.jpg>" \
  ... \
  --prompt "$(cat 即梦提示词_提交版.txt)" \
  --duration 15 \
  --ratio 9:16 \
  --model_version seedance2.0fast \
  --video_resolution 720p
```

**时长设置规则：**
- `--duration` 必须设置为所有镜头时长之和（如 3+4+4+4=15）
- 禁止使用固定5秒，必须根据分镜脚本计算实际总时长

**参考图选择原则：**
- 从分镜脚本中**所有镜头**的"参考图"字段提取所有不重复的参考图
- 参考图路径必须与实际下载的文件名一致
- `--image` 参数最多支持9张图片（需去重）
- 如果总参考图超过9张，选择每镜最重要的1-2张

**注意事项：**
- **必须使用 `multimodal2video` 而非 `image2video`**，前者支持全能参考和多图上传
- `--duration` 支持 4-15 秒，必须等于所有镜头时长之和
- `--ratio` 必须显式指定为 `9:16`（竖屏）
- `--model_version` 必须指定为 `seedance2.0fast`
- 如果遇到 `ExceedConcurrencyLimit` 错误，等待60秒后重试
- 提交成功后，记录 `submit_id` 到 `dreamina_task.json`

```bash
~/.local/bin/dreamina download --submit_id <submit_id> --output_path "<run目录/full_video.mp4>"
```

## 错误处理

| 场景 | 处理方式 |
|------|----------|
| 商品信息字段缺失 | 停止流程，列出缺失字段，请用户补充 |
| 即梦CLI不可用 | 提示用户手动在官网生成，提供提示词，询问文件路径 |
| 即梦CLI执行失败 | 记录错误信息，请用户确认CLI状态，尝试回退手动模式 |
| 即梦返回长队列/预计等待过长 | 立即停止自动轮询，写入 `dreamina_task.json`，通知用户后续人工触发下载 |
| duration报错 | 调整 `--duration` 在 4-15 秒合法范围内重试 |
| 视频文件未找到 | 询问用户视频文件位置，尝试常见下载目录 |
| 目录已满/写入失败 | 告知用户，检查目录权限 |
