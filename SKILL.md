# 生鲜商品视频策划技能

## 固定配置

```
视频输出根目录：~/freshVedioPlanning/
视频规格：时长 ≤15s | 分辨率 720p | 比例 9:16
即梦CLI命令：~/.local/bin/dreamina multimodal2video（全能参考模式，支持多图）
```

## 触发条件

每次任务开始前，**逐项收集商品信息**（见下方清单），如有缺失立即暂停并请求用户提供。不得跳过，不得自行搜索或推理补充。

## 第一步：收集商品信息

向用户收集以下字段（其中 `product_analysis` 为选填，其余为必填）：

| 字段 | 说明 | 示例 |
|------|------|------|
| `product_name` | 商品名称 | 美汁源益生元凤梨百香果复合果汁饮料 |
| `sku` | 商品SKU编号 | 556127 |
| `selling_points` | 商品核心卖点（多条，用换行分隔） | 30%膳食纤维、凤梨+百香果双果、0脂 |
| `user_demands` | 用户需求/使用场景（多条） | 餐后解腻、肠道健康、下午茶 |
| `hot_topics` | 当前热点/季节机会（可选填） | 轻体季、健康饮品 |
| `product_analysis` | 商品分析（选填，可粘贴类目表现/搜索趋势/行业调研/价格力/竞争力等结论） | 细分类目表现、搜索热度、行业分析、价格力、竞争力、总分 |
| `main_image` | 商品主图本地路径 | ~/freshVedioPlanning/.../main_image.jpg |
| `detail_images` | 商详图片路径列表（至少1张） | ~/freshVedioPlanning/.../detail1.jpg |

**缺少任意必填字段 → 立即告知用户缺失项，停止后续流程。`product_analysis` 为选填，不作为阻断项。**

## 第二步：建立会话目录

```bash
python3 ~/.qclaw/skills/fresh-video-planning/scripts/init_session.py \
  --product "商品名称" --sku "sku编号" --root "~/freshVedioPlanning"
```

## 第三步：切入点分析（世界观构建）

将分析结果保存至会话目录的 `01_切入点分析.md`。

若提供了 `product_analysis`，先提取其中可直接用于创意表达的“机会信号”（类目竞争、季节趋势、价格带、人群认知），仅允许引用有明确依据的结论，禁止杜撰数据。

必须输出以下结构，不得省略：

### 3.1 产品诊断（5维）

围绕商品信息提取 5 个维度：

1. 原料（产地/品种/独特性）
2. 成分（健康数据/核心营养）
3. 颜色（天然色泽来源）
4. 香气（气味记忆点）
5. 工艺（处理方式/NFC/冷灌等）

### 3.2 卖点翻译（核心）

将每个卖点翻译成可视化角色或场景，禁止直白罗列参数：

- 原料 -> 产地奇观（庄园、矿区、果园、农场）
- 成分 -> 微观角色（粒子、晶体、能量线）
- 颜色 -> 巨物奇观（切开瞬间、渐变流体）
- 香气 -> 可视化氛围（烟雾、光丝、雾气）
- 工艺 -> 游戏化流程（漂流、汇合、闯关）

新增“真实卖点提取”步骤（必须执行）：

1. 从主图、商详图、卖点文案中提取“可被证据支撑”的卖点清单（禁止凭空补充）。
2. 为每个卖点标注类型：  
   - A 类：产品外观/包装可直接验证  
   - B 类：原材料特性可直接验证  
   - C 类：功能或口感主张（需谨慎表达，不可伪造内部状态）
3. 仅允许 A/B 类卖点直接转镜头；C 类卖点通过场景结果、情绪反馈、抽象质感间接表达。
4. 若核心卖点属于“原材料特性”，分镜前两镜必须表现原材料特点，不得与产品内部形态融合。

### 3.3 世界观构建（只保留 1 套）

基于上一步，构建一个 AI 原生世界观（只产出 1 套）：

- 世界设定：产品发生在什么奇观场域（例：味觉游乐场、原料冒险基地）
- 角色设定：原料/成分在世界中的角色身份
- 目标设定：产品如何作为“冒险终点/价值兑现”
- 核心反差：先破旧认知，再给新认知（例如“以为人工调色，实际天然来源”）

### 3.4 平台适配约束（抖音/即梦）

在分析末尾给出“本次必须遵守”的 4 条硬约束：

1. 前1秒必须出现视觉主体（果实/瓶身/动作）
2. 前3秒必须出现冲突或反差（尺度、空间、情绪）
3. 每4秒至少一次信息增量（新动作/新角度/新卖点）
4. 结尾2秒必须完成品牌落版 + 记忆锚点（标签/口号/动作）

## 第五步：生成分镜脚本（单套）

生成1套完整分镜脚本，保存至 `03_分镜脚本.md`。

若存在 `product_analysis`，分镜中的卖点文案与镜头重点应优先对齐“高置信分析结论”（如搜索高热词、已验证差异点、价格力优势）；未被分析支持的卖点，需降级为辅助表达，不可作为核心钩子。

### 5.1 分镜脚本生成逻辑

必须按以下顺序生成，且只产出 1 套可直接执行的后半段产品成片脚本：

1. **先做素材事实表**：逐张写明“文件名 -> 实际内容 -> 可用于哪一镜”，禁止凭文件名猜图。  
2. **再写世界观与质感基准**：给出一句核心世界观 + 画幅/质感红线。  
3. **再做参考图分工**：每镜至少声明 2 种角色（首帧锚定/主体纹理/色彩氛围/构图关系），避免“同一张图承担全部任务”。  
4. **按镜头逐个展开**：建议 4 镜（3-4-4-4 秒或近似），每镜必须含构图公式、光影公式、色彩公式。  
5. **每镜补齐执行参数**：必须写清镜头运动、时间事件（如 T+2.0s）、转场方式、对焦策略、本镜禁止项。  
6. **每镜必须带 Seedance Prompt**：镜头描述和最终提示词一一对应，避免“脚本写一套、提示词是另一套”。  
7. **最后补时间轴和素材分配汇总**：确保总时长 <= 15 秒，素材用途可追溯。  
8. **禁用不实内部镜头**：无法证实包装/杯内状态时，不得写拆封后内部细节。

## 第六步：视频生成（调用即梦CLI图生视频）

### 6.1 调用即梦前的提示词生成逻辑

#### 6.1.1 生成逻辑

将第五步确认后的分镜脚本压缩为“可直接贴入 `multimodal2video --prompt` 的连续文本”，规则如下：

1. **按 Shot 顺序拼接**：`Shot01` 到 `Shot04/05`，每段都包含“参考图 + 连续描述”。  
2. **镜头字段进 prompt**：画面内容、运镜、光线、色彩、品牌露出写进同一段自然语言，且必须包含镜头执行参数。  
3. **补时间锚点**：每镜至少 1-2 个时间事件（如 T+2.0s），用于控制元素出现时机。  
4. **保留核心叠字信息**：只保留关键卖点字样，写清出现时点、位置、样式，禁止堆砌无效字体参数。  
5. **保持可执行性**：避免抽象判断词，必须写清主体/动作/环境/镜头运动/转场/对焦。  
6. **镜头级负向约束**：每镜均要写禁止项（尤其是内部展示相关），并给出安全替代表达。

#### 6.1.4 提交前确认（保持强制）

**严格流程**：生成提示词 -> 写入手工确认 artifact -> 展示给用户 -> 用户确认后才调用即梦 CLI。

每次提交前，向用户展示：
1. 完整即梦提示词（可复制）
2. 预计积分消耗
3. 参考图选择与用途
4. 创意保真检查（开场钩子/中段推进/结尾落版）

用户回复“可以/好/确认”后，才执行 `dreamina multimodal2video`。

### 6.2 调用即梦CLI（全能参考模式）

**关键：使用 `multimodal2video` 命令启用“全能参考”模式，支持上传多张图片作为参考。主图必须入选，商详图按需追加。一次提交生成完整视频（不分镜提交）。**

```bash
~/.local/bin/dreamina multimodal2video \
  --image "<会话目录main_image.jpg路径>" \
  --image "<商详图1路径>" \
  --image "<商详图2路径>" \
  --prompt "<用户确认后的即梦提示词>" \
  --duration <总秒数，4-15之间> \
  --ratio 9:16 \
  --model_version seedance2.0fast \
  --video_resolution 720p \
  --poll 60
```

**注意事项：**
- **必须使用 `multimodal2video` 而非 `image2video`**，前者支持全能参考和多图上传，后者仅支持单张首帧图
- `--image` 参数可重复多次，最多支持9张图片（主图必选 + 商详图按需追加）
- 主图必须入选参考图；商详图是否入选需基于“镜头动作是否需要补充细节/材质/功能展示”判断，不做固定限制
- 在提交前，先完成“主图+商详图内容分析”，再结合最终运镜脚本确定参考图清单
- `--duration` 支持 4-15 秒（seedance2.0fast）
- `--ratio` 必须显式指定为 `9:16`（竖屏），multimodal2video 不会自动推断比例
- 如果总时长超过15秒，分多次提交，或截取核心镜头
- 如果 `--duration` 报错"should >=4 && <=15"，改为合法值
- `--poll 60` 仅用于“快速拿到首个状态”，不用于长时间持续等待
- 任务提交成功后会返回 `submit_id`，必须写入会话目录的 `即梦任务.json` 以便后续人工下载

### 6.3 长队列处理与任务记录（替代持续轮询）

高峰期视频生成可能超过2小时，禁止长时间自动轮询。

**执行策略（严格按顺序）：**
1. 提交 `multimodal2video` 后，最多进行一次短轮询（`--poll 60`）获取初始状态。
2. 若状态为排队中/处理中，且提示等待时间较长（例如预计超过30分钟或明显处于高峰队列），立即停止自动等待。
3. 将任务信息写入 `即梦任务.json`，至少包含：
   - `submit_id`
   - `status`（queueing/processing/success/failed）
   - `created_at`
   - `last_checked_at`
   - `note`（如“高峰期长队列，改为人工触发下载”）
4. 向用户明确告知：任务已成功提交并记录，可稍后人工触发查询/下载。
5. 本轮流程到此结束，不再循环 `query_result`。

`即梦任务.json` 示例：
```json
{
  "submit_id": "xxxxxxxx",
  "status": "queueing",
  "created_at": "2026-04-20T10:00:00+08:00",
  "last_checked_at": "2026-04-20T10:01:00+08:00",
  "note": "高峰期长队列，改为人工触发下载"
}
```

### 6.4 人工触发查询与下载

```bash
# 查询状态
~/.local/bin/dreamina query_result --submit_id <submit_id>

# 等待完成后下载（当 gen_status=success 时）
~/.local/bin/dreamina download --submit_id <submit_id> --output_path "<run目录/完整视频.mp4>"
```

推荐使用一键脚本（读取并回写 `即梦任务.json`）：
```bash
python3 ~/.qclaw/skills/fresh-video-planning/scripts/manual_download.py \
  --run-dir "<run目录路径>"
```

只查询不下载：
```bash
python3 ~/.qclaw/skills/fresh-video-planning/scripts/manual_download.py \
  --run-dir "<run目录路径>" \
  --query-only
```

人工触发下载成功后，需要同步更新 `即梦任务.json`（状态、最后检查时间、下载路径）。

## 错误处理

| 场景 | 处理方式 |
|------|----------|
| 商品信息字段缺失 | 停止流程，列出缺失字段，请用户补充 |
| 即梦CLI不可用 | 提示用户手动在官网生成，提供提示词，询问文件路径 |
| 即梦CLI执行失败 | 记录错误信息，请用户确认CLI状态，尝试回退手动模式 |
| 即梦返回长队列/预计等待过长 | 立即停止自动轮询，写入 `即梦任务.json`，通知用户后续人工触发下载 |
| duration报错 | 调整 `--duration` 在 4-15 秒合法范围内重试 |
| 视频文件未找到 | 询问用户视频文件位置，尝试常见下载目录 |
| 目录已满/写入失败 | 告知用户，检查目录权限 |
